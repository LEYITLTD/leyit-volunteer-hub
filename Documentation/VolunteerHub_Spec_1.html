Volunteer login portal — what volunteers can do once logged in
  • View their current compliance status and any outstanding actions (e.g. "Upload your DBS certificate to complete registration")
  • Upload or re-upload their DBS certificate at any time without contacting admin
  • Browse published events and apply for available roles
  • View their personal QR code for event check-in (only visible once confirmed for an event)
  • Track their points balance, current reward tier and full points transaction history
  • Update profile — phone, address, emergency contact, dietary requirements, medical info
Register→ Upload DBS→ Admin approves DBS→ Compliance approved→ Welcome email sent→ Can apply to events
Compliance — DBS & Refinitiv World-Check

Both checks are done once at registration and reused across all events. Neither check needs to be repeated unless a DBS certificate expires or a Refinitiv flag changes. A volunteer cannot check in to any event unless both checks pass.

① DBS Check — manual admin review
  • Volunteer uploads their DBS certificate (PDF or image) at registration or later via their login portal
  • Admin is notified in the Compliance tab — document is visible in the dashboard
  • Admin reviews and clicks Approve or Reject
  • Admin enters the DBS expiry date at point of approval
  • Automated reminder emails sent at 60 and 30 days before expiry
  • On expiry, status reverts to Pending until renewed and re-approved
  • All decisions logged with admin name and timestamp
② Refinitiv World-Check (LSEG) — API auto-screen
  • System automatically calls the LSEG World-Check One API the moment a volunteer submits their registration
  • API is passed: full name, date of birth, nationality
  • Response is one of: CLEAR, POSSIBLE_MATCH, or HIGH_RISK
  • CLEAR → auto-passed, no admin action needed, marked green in dashboard
  • POSSIBLE_MATCH / HIGH_RISK → flagged in the Compliance panel for manual review
  • Admin clicks Review to see match details (name, category, risk level, source), then either clears ("not same person") or rejects the volunteer
  • All Refinitiv decisions logged with admin name and timestamp for audit

Multi-Event Management
Event lifecycle
Admin creates event→ Sets roles + capacities→ Saves as Draft→ Publishes → email to all approved volunteers→ Volunteers apply→ Admin confirms applications→ Event day (Active)→ Completed + post-event email
Admin can
  • Create events with name, date/time, venue, description
  • Add multiple roles (Ushers, Welcome Team, Packaging, etc.) each with a capacity limit
  • Set bonus QR station time windows per role
  • Publish to notify all approved volunteers by email
  • View and manage all applications per event, filtered by role or status
  • Confirm, decline or waitlist applications
  • See real-time attendance on event day
Volunteers can
  • See all published events in their portal
  • Apply to a specific event and select a preferred role
  • View their confirmation status and personal QR code
  • Cancel their application (frees the spot for the next waitlisted person)
  • See how many points they earned at each past event
Waitlist logic
  • When a role is at capacity, new applications are automatically waitlisted with a position number stored in the database
  • When a confirmed volunteer cancels, the next person on the waitlist for that role is auto-notified by email with a link to confirm their place
  • They have 48 hours to confirm before it moves to the next person (configurable)

Pre-event email blast — admin-composed from the web app
  • Admin selects an event, then clicks "Send email to confirmed volunteers"
  • Composes subject line and body in a rich-text editor built into the dashboard
  • Attaches one or more PDF files by uploading directly in the composer (stored in Cloudflare R2, sent as email attachments via Resend)
  • A preview shows the full email and recipient count before sending
  • Sends only to volunteers with status = Confirmed for that event
  • Full send log stored: subject, timestamp, attachment filenames, recipient count
  • All past campaigns visible in an Email History section per event
PDF files are uploaded to Cloudflare R2 storage, which generates a signed URL. Resend then attaches the file to the outgoing email. The volunteer receives a genuine email attachment — no links to click.
QR Code Check-in
Entry check-in
  • Each confirmed volunteer receives a unique QR code by email as a PNG attachment
  • QR encodes a secure UUID token unique to that volunteer + event combination
  • Admin scans at the door using the Check-in tab on any phone or tablet
  • System validates: compliance = Approved, application status = Confirmed, token is authentic
  • Successful scan → checked in + 100 attendance points awarded
  • Duplicate scan → blocked with "Already checked in" message
  • Volunteer without approval → denied entry, message shown to admin
Bonus QR stations
  • 4 self-service stations placed around the venue — Setup, Packaging, Social Media, Cleanup
  • Each station has a different QR code encoding its station type
  • Volunteer scans with their phone camera
  • System checks: (a) entry check-in already completed, (b) current time is within the station's valid window, (c) not already claimed for this event
  • Outside time window → scan rejected, no points
  • Cleanup station enforces sign-out: only valid after event end time
Points System & Time Logic
Points categories
CategoryPointsTriggerTime window logic
Early bird +50 Application confirmed Only awarded if confirmed_at ≤ event_date − 14 days (configurable per event)
Attendance +100 Entry QR scan Valid from doors_open to event_start + 90 min

Time window enforcement rules
  • Each bonus station has station_window_start and station_window_end stored in event_roles — set by admin when creating the event
  • At scan time, server checks: now() BETWEEN window_start AND window_end
  • Outside the window → scan rejected: "This station is not currently active"
  • Entry check-in must exist before any bonus station points can be awarded
  • Points are deduplicated: each category can only be earned once per volunteer per event
Reward tiers & ledger
  • Every award and deduction is written to points_transactions — append-only, cannot be edited
  • Total points = SUM(amount) per volunteer, calculated in real time
  • Certificate at 100+ pts
  • Silver badge at 250+ pts
  • Gold badge at 400+ pts
  • Admin can manually award bonus points or deductions with a written reason
  • Congratulations email sent automatically when a volunteer crosses a tier threshold

Security
  • DBS certificates stored in a private R2 bucket — never publicly accessible; served via signed URLs that expire
  • All API keys stored as environment variables — never in code
  • Role-based access control: volunteers cannot access admin routes; coordinators cannot access compliance data
  • Supabase row-level security (RLS) ensures volunteers can only query their own records at the database level
  • HTTPS everywhere — Vercel provides automatic SSL
  • Admin activity log is append-only and cannot be edited or deleted
