# Architectural Decision Record — VolunteerHub

> Append-only log. Each decision has a status: **Accepted** | **Superseded** | **Deprecated**.

---

## ADR-001 · Next.js as the unified frontend + API framework
**Date:** 2026-06-17  
**Status:** Accepted

**Context:** The platform needs both an admin dashboard and a volunteer portal, plus server-side API routes for auth, compliance, QR scanning, and email triggers.

**Decision:** Single Next.js 16 (App Router) codebase. Admin and volunteer portals live under `/admin/**` and `/volunteer/**` route groups. API routes under `/api/**`.

**Consequences:** No separate Express server to maintain. Server Components for data fetching, Client Components for interactivity. Route group layouts enforce role-specific shells (sidebar vs top-nav).

---

## ADR-002 · Supabase for Postgres + Auth + Row-Level Security
**Date:** 2026-06-17  
**Status:** Accepted

**Context:** Needed managed Postgres, built-in auth (email/password + JWTs), and row-level access control without rolling a custom auth system.

**Decision:** Supabase (Pro tier recommended — prevents project pausing on free tier). Supabase Auth manages volunteer and admin sessions. RLS policies ensure volunteers can only read their own rows at the DB level, regardless of API behaviour.

**Consequences:** Auth and data in one platform. `@supabase/ssr` used for server-side sessions in Next.js middleware and Server Components. Admin service-role key stored only in server-side env vars and never exposed to the browser.

## ADR-003 · Cloudflare R2 for DBS certificates and email PDF attachments
**Date:** 2026-06-17  
**Status:** Accepted

**Context:** DBS certificate uploads and pre-event PDF attachments need secure storage with no egress fees at ~700 volunteers' scale.

**Decision:** Cloudflare R2 (S3-compatible API). Documents stored in a private bucket. Files served exclusively via signed URLs that expire (15 minutes for DBS certificates). Admin never links volunteers to their raw bucket path.

**Consequences:** Zero egress cost. ~700 DBS PDFs + event attachments ≈ 2–3 GB, well within the 10 GB free tier. S3-compatible so the `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner` packages handle signing.

## ADR-004 · Resend (Pro) for all transactional and campaign email
**Date:** 2026-06-17  
**Status:** Accepted

**Context:** Event-day confirmation bursts (~100 emails simultaneously) exceed Resend free tier's 100/day cap. All 10 automated triggers need reliable delivery.

**Decision:** Resend Pro (~£16/month, 50k emails/month). React Email templates compiled server-side and sent via Resend's REST API. QR codes attached as PNG buffers generated by `qrcode` (npm). PDF attachments fetched from R2 signed URLs and streamed to Resend.

**Consequences:** No daily cap. 50k/month is ~25× actual volume. Open and click tracking via Resend dashboard. `resend_batch_id` stored per campaign for webhook-based delivery receipts.

## ADR-005 · Tailwind CSS v4 with CSS custom-property token system
**Date:** 2026-06-17  
**Status:** Accepted

**Context:** The design uses a named token set (Nūr theme: espresso chrome, gold accent, warm off-white). Inline styles in prototypes must become reusable, refactorable utilities.

**Decision:** All design tokens live as CSS custom properties in `:root` inside `globals.css`. The `@theme inline` block maps them to Tailwind v4 utilities (`bg-gold`, `text-text-secondary`, `rounded-lg` etc.). No inline `style={{}}` attributes except for dynamic values (e.g. progress bar widths). No hardcoded hex colours in component files.

**Consequences:** Single source of truth for the colour palette. Theme changes require editing one file. Tailwind's JIT generates only used utilities.

## ADR-006 · Cormorant Garamond (display) + Hanken Grotesk (UI) via next/font
**Date:** 2026-06-17  
**Status:** Accepted

**Context:** The Nūr brand uses Cormorant Garamond (serif calligraphy feel) for page headings and Hanken Grotesk (modern grotesque) for all UI text, echoing the calligraphic logo aesthetic.

**Decision:** Both fonts loaded via `next/font/google` in the root layout, injected as CSS variables (`--font-cormorant`, `--font-hanken`). The `@theme` block maps them to `font-display` and `font-ui` Tailwind utilities. No `<link rel="stylesheet" href="fonts.googleapis.com/...">` in HTML — eliminates the render-blocking request.

**Consequences:** Font files self-hosted via Next.js (zero external network request at runtime). Subset to Latin. `font-display: swap` prevents invisible text during load.

## ADR-007 · Compliance requires BOTH DBS approval AND Refinitiv clear before event access
**Date:** 2026-06-17  
**Status:** Accepted

**Context:** Safeguarding requirement: a volunteer with an approved DBS but a flagged Refinitiv screening must not be able to check in to events.

**Decision:** `overall_status` on `volunteer_compliance` is computed from both checks via a DB trigger / server-side logic. The QR check-in scanner validates `overall_status = 'approved'` in real time — individual check statuses are not re-evaluated at scan time.

**Consequences:** Single field to gate event entry. Trigger keeps it consistent even if direct DB edits are made. Audit log records every status change with actor and timestamp.

## ADR-008 · Points are append-only transactions, never edited
**Date:** 2026-06-17  
**Status:** Accepted

**Context:** Points history must be auditable. Corrections must be transparent (add a deduction row) not invisible (edit an existing row).

**Decision:** `points_transactions` table has no UPDATE or DELETE RLS policies — inserts only. Total balance is always `SUM(amount)` computed at query time. Manual corrections use `type = 'deduction'` with a required description. Supabase RLS enforces this at the database level.

**Consequences:** Full immutable audit trail. Admin UI shows all transactions including negative ones. `deduction` rows appear with red amounts in volunteer history.

## ADR-009 · QR tokens are per-volunteer-per-event UUIDs
**Date:** 2026-06-17  
**Status:** Accepted

**Context:** Entry QR codes must be unique, non-transferable, and scannable by any phone. Bonus station QR codes are static and encode the station type.

**Decision:** Entry QR: `qr_token` UUID stored in `event_applications`, encoded into a QR PNG via `qrcode` (server-side), emailed as an attachment. Bonus station QRs: static QR codes encoding a URL like `/checkin/station/setup?event_id=<uuid>`, placed physically at stations. Server validates time window and deduplication on every scan.

**Consequences:** Entry tokens are one-time-use per event (duplicate scan blocked). Bonus station URLs are not secret — they're publicly scannable — but the server validates the volunteer's identity (session cookie) and time window server-side.

## ADR-010 · GitHub → Vercel CI/CD; code written on local machine only
**Date:** 2026-06-17  
**Status:** Accepted

**Context:** All code is authored locally (this machine). Cloud environments are GitHub (source), Vercel (hosting), and Supabase (database).

**Decision:** Single `main` branch deploys to Vercel production. Feature branches create Vercel preview deployments automatically. Supabase migrations run via `supabase db push` from local machine after review. Secrets (Supabase keys, Resend API key, R2 credentials) stored in Vercel environment variables — never committed.

**Consequences:** No CI runner needed for now. One-click Vercel rollbacks available. `.env.local` holds secrets for local dev and is gitignored.

## ADR-011 · Supabase Pro tier over Free for production reliability
**Date:** 2026-06-17  
**Status:** Accepted

**Context:** Supabase free projects pause after 1 week of inactivity. VolunteerHub has multi-week gaps between events when activity is low.

**Decision:** Use Supabase Pro (£20/month). Includes: no pausing, daily automated backups, point-in-time recovery, higher connection limits.

**Consequences:** Monthly infra cost rises to £36/month (Supabase Pro £20 + Resend Pro £16). Annual cost £432/year. Acceptable for a platform managing 700 volunteers and £45k–70k of development investment.
